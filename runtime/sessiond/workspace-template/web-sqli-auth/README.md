# 题目说明：登录逻辑与输入信任边界

## 题目背景

某业务系统提供了一个简化的登录接口。开发者希望通过用户名和密码判断用户身份，但登录逻辑对用户输入的处理存在问题。你的任务是从接口行为出发，验证这个问题是否会导致认证绕过，并解释根因与修复方式。

本题不要求扫描整个站点，也不要求枚举真实用户。重点是观察登录请求、构造测试输入、判断认证逻辑是否被输入内容影响。

## 目标服务

目标服务地址：

```text
http://127.0.0.1:18080
```

终端里也可以直接使用环境变量：

```bash
echo "$TARGET_BASE_URL"
```

可访问接口：

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `GET` | `/healthz` | 检查目标服务是否在线 |
| `POST` | `/login` | 提交用户名和密码进行登录 |

`/login` 使用表单格式接收参数：

| 参数 | 含义 |
| --- | --- |
| `username` | 用户名 |
| `password` | 密码 |

普通错误账号通常会返回 `401 Unauthorized`，响应体中 `ok` 为 `false`。如果你验证出认证绕过，响应会变成成功状态，并返回管理员身份相关字段。

## 建议操作顺序

1. 确认目标服务在线。
2. 发送一次普通错误账号登录请求，记录状态码和响应体。
3. 分别改变 `username` 和 `password` 的输入内容，观察响应是否发生异常变化。
4. 找到能说明认证逻辑被绕过的最小请求。
5. 回到网页右侧“提交”区域，写清楚根因、验证过程和修复建议。

## 示例起步命令

确认服务在线：

```bash
curl -i "$TARGET_BASE_URL/healthz"
```

发送普通错误账号请求：

```bash
curl -i -X POST "$TARGET_BASE_URL/login" \
  -d "username=alice&password=wrong"
```

你可以继续用 `curl` 构造不同的 `username` 和 `password` 值。题目不会要求使用图形化浏览器，终端里的 `curl` 和 `python3` 已足够完成验证。

## 提交要求

提交答案应包含三部分：

1. **根因说明**：登录逻辑为什么会被用户输入影响。
2. **验证过程**：你发送了什么类型的请求，正常失败和异常成功分别有什么现象。
3. **修复建议**：应该如何改造查询或认证逻辑，避免同类问题再次出现。

不要只写“完成了”或只贴命令输出。答案需要能让教师看出你理解了输入信任边界、认证绕过影响和修复方向。
